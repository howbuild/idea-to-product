# PRD: 요청 관리 게시판

## 1. 요약

요청 관리 게시판은 사용자가 요청을 등록하고, 담당자가 목록에서 확인한 뒤 상태를 관리하는 MVP입니다.

## 2. 배경

요청이 대화나 메모에 흩어지면 담당자 확인, 상태 공유, 완료 여부 추적이 어렵습니다.

## 3. 문제 정의

사용자는 요청이 접수되었는지 알기 어렵고, 담당자는 처리해야 할 요청을 한곳에서 보기 어렵습니다.

## 4. 목표

- 요청 등록과 확인을 한 화면 흐름으로 정리합니다.
- 요청 상태를 `접수`, `처리중`, `완료`, `보류`로 관리합니다.
- 완료까지 걸린 시간을 측정할 수 있게 합니다.

## 5. 핵심 사용자

- 요청 등록 사용자
- 요청 처리 담당자

## 6. 사용자 여정

1. 사용자가 요청을 작성합니다.
2. 사용자가 요청을 등록합니다.
3. 담당자가 요청 목록을 확인합니다.
4. 담당자가 상태를 변경합니다.
5. 사용자가 현재 상태를 확인합니다.

## 7. 핵심 유저 플로우

### 진입과 다음 행동

사용자는 요청 작성 화면에서 요청을 등록하고, 등록 후 요청 목록이나 현재 상태 확인 화면으로 이동합니다.

### 사용 규모

요청 등록 수, 담당자 확인 비율, 상태 변경 수를 통해 요청 처리량과 병목 여부를 확인합니다.

### 품질 기준

사용량이 늘어나도 잘못된 상태 변경과 개인정보 기록 위험을 줄이기 위해 권한정책과 기록 정책을 둡니다.

## 8. MVP 범위

- 요청 등록
- 요청 목록 확인
- 상태 필터
- 검색
- 빈 상태
- 상태 변경
- 상태 변경 기록 정책 결정
- 완료까지 걸린 시간 측정

## 9. 제외 범위

- 실시간 채팅
- 복잡한 권한 관리
- 자동 배정
- 외부 시스템 연동

## 10. 핵심 기능

- 요청 등록 폼
- 요청 목록
- 검색
- 상태 필터
- 상태 변경
- 빈 상태 표시

## 11. 화면 구성

- `request-create-form`: 요청 등록 영역
- `request-list-section`: 요청 목록
- `request-search-input`: 검색
- `request-status-filter`: 상태 필터
- `empty-request-state`: 빈 상태
- `request-status-change`: 상태 변경

## 12. 주요 정책 요약

- `POL-001`: 요청 등록은 로그인한 사용자에게 허용합니다.
- `POL-002`: 상태 변경은 담당자에게만 허용합니다.
- `POL-003`: 상태 변경 기록은 남기는 방향으로 검토합니다.

## 13. 측정과 기록 요약

- `MEA-001`: 요청 등록 이벤트
- `MEA-002`: 상태 변경 로그
- `MEA-003`: 완료까지 걸린 시간

## 14. 성공 지표

- 등록된 요청 수
- 담당자가 확인한 요청 비율
- 요청 완료까지 걸린 시간

## 15. 리스크

- 상태 변경 권한이 불명확하면 잘못된 변경이 생길 수 있습니다.
- 기록 범위가 과하면 운영 부담이 늘 수 있습니다.

## 16. 미결정 사항

- 상태 변경 사유를 필수로 받을지
- 요청 삭제를 MVP에 포함할지
