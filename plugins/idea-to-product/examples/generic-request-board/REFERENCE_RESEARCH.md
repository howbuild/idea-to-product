# REFERENCE_RESEARCH: 요청 관리 게시판

## 1. 조사 목적

요청 관리 게시판의 상태값, 권한정책, 측정/기록 후보를 만들기 위한 참고 패턴을 정리합니다.

## 2. 조사 대상

외부 웹 조사는 수행하지 않았습니다. 일반적인 요청 관리 업무 흐름을 예시로 사용합니다.

## 3. 참고한 출처

| 출처 | 확인 날짜 | 관찰한 내용 | 사실/추론 | 한계 |
|---|---|---|---|---|
| 사용자가 제공한 일반 제품 흐름 | 2026-05-25 | 요청 등록, 목록 확인, 상태 변경이 필요함 | 추론 | 실제 경쟁제품 조사는 미수행 |

## 4. 관찰한 공통 패턴

- 요청은 상태값으로 관리됩니다.
- 담당자만 상태를 변경할 수 있는 정책이 필요합니다.
- 완료까지 걸린 시간을 측정할 수 있습니다.

## 5. 현재 제품에 적용 가능한 부분

- `접수`, `처리중`, `완료`, `보류` 상태값
- 상태 변경 권한정책
- 상태 변경 로그

## 6. MVP에는 과한 부분

- 실시간 채팅
- 자동 배정
- 복잡한 관리자 기능

## 7. 정책 후보

- 요청 등록 권한
- 상태 변경 권한
- 상태 변경 기록 정책

## 8. 상태값 후보

- 접수
- 처리중
- 완료
- 보류

## 9. 권한정책 후보

- 로그인 사용자는 요청 등록 가능
- 담당자는 상태 변경 가능

## 10. 측정/기록 후보

- 요청 등록 이벤트
- 상태 변경 로그
- 완료까지 걸린 시간

## 11. 사용자에게 물어볼 질문

- 사용자는 어디서 요청 등록 화면으로 오고, 등록한 뒤 어디로 가야 하나요?
- 요청 등록이 많아질 때 어디에서 가장 많이 막힐 수 있나요?
- 사용자가 많아져도 잘못된 상태 변경을 막으려면 어떤 권한정책이 필요할까요?
- 상태 변경 사유를 필수로 받을까요?
- 상태 변경 기록은 어느 수준까지 남길까요?

## 12. 조사 한계

외부 경쟁제품 조사는 하지 않았습니다. 확정 요구사항이 아니라 질문 후보입니다.
