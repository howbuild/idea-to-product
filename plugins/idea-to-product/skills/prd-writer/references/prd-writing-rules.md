# PRD Writing Rules

- Write only confirmed decisions as requirements.
- Mark assumptions and unresolved items clearly.
- Mark each feature requirement as `Must-have` or `Nice-to-have`.
- Mark operating intent for each requirement: `Production`, `Pilot`, `Demo`, `Test`, or `One-off`.
- Mark scalability for each requirement.
- Do not write Nice-to-have items as confirmed Must-have requirements.
- Include policy and measurement summaries.
- In PRD summaries, use natural language instead of exposing `POL-*` or `MEA-*` IDs.
- Use plain expression with professional product wording.
- Do not leave interview questions in PRD sections; state confirmed decisions or mark unresolved items.
- Avoid filler and AI-like openings.
