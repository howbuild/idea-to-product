# Reference Discovery Workflow

## Role

After an idea is entered, check whether similar products or comparable work flows can improve the next questions.

This is not market research for its own sake. Reference discovery exists to create better questions, requirement options, policies, states, measurement candidates, Must-have/Nice-to-have judgments, and operating-intent judgments.

## Execution Timing

- after Core Intent Card creation
- before the first structural wireframe
- when the user asks "다른 서비스는 어떻게 해?"
- when no domain pack exists and question candidates are weak
- when Must-have priority is ambiguous

## Steps

1. Identify the current idea's product type.
2. Find comparable products, work flows, or domain practices when tools and access allow it.
3. Extract common patterns without copying a competitor.
4. Convert common patterns into question candidates.
5. Check fit against Core Intent and Must-have/Nice-to-have priority.
6. Record results in `REFERENCE_RESEARCH.md` or `DECISION_LOG`.

## Output

- 참고한 제품/업무 흐름
- 관찰한 공통 패턴
- 우리 제품에 적용 가능한 부분
- Nice-to-have 후보
- 운영 의도/확장 가능성 판단
- 추가 질문 후보
- 정책 후보
- 상태값 후보
- 측정/기록 후보
- 출처
- 확인 날짜
- 조사 한계

## If Research Is Not Possible

If Browser, Web, MCP, or document access is unavailable:

- use only references provided by the user
- mark unverified parts as 미확인
- do not invent competitor facts
